# 코드리뷰 (1) : 왜 코드 리뷰를 해야 할까?

## 시작하며

1년이 조금 넘는 기간 동안 팀원들과 코드 리뷰를 진행했다. 입사했을 당시에는 코드 리뷰가 활발히 진행되지 않았고 체계가 잡혀있지 않았다. 그렇다 보니 시간이 지날수록 코드 리뷰를 안 하는 것이 당연시되었다. 그래서 팀원들과 코드 리뷰에 대한 회의를 하였고, 지금은 완전히 팀 문화에 정착하여 지속적으로 코드 리뷰를 해오고 있다.

만약 아직 코드 리뷰를 시작하지 않은 팀이 있다면, 코드 리뷰를 시작하는 데 도움이 될 수 있도록 1년간의 경험을 통해 배운 것을 두 개의 질문으로 나눠서 답을 정리하고 공유하려고 한다. 첫 번째는 “왜 코드 리뷰를 해야 할까?”이며 두 번째는 “어떻게 코드 리뷰를 해야 할까?”이다.

그중 이 글은 “왜 코드 리뷰를 해야 할까?”에 대한 내용을 정리한 것이다. 취업하기 전, 여러 명의 사람들과 프로젝트를 할 때 주위에서 코드 리뷰하는 것이 좋다고 하니 따라서 했던 적이 있다. 프로젝트가 끝났을 때 “코드 리뷰가 진짜 좋은 것이 맞아?”라는 의문이 들었는데 입사 후부터 지금까지 코드 리뷰를 진행해 보니 실질적으로 긍정적인 면을 몸소 느낄 수 있었다.

## 1. 팀원이 부재 중일 때

대부분의 프로덕트는 여러 명의 개발자가 task를 나눠서 개발을 진행한다. 혼자 모든 개발을 책임진다는 것은 많은 위험부담이 따른다. 그 한 명이 부재 중일 때 개발이 진행되지 못하고 에러가 발생했을 때 대응이 늦어진다면 사용자들이 많이 이탈할 수도 있다. 하지만 여러 명의 개발자가 있다고 하더라도 서로의 task에 대해 이해하지 못하면 혼자 개발하는 것과 크게 다르지 않다.

같은 프로덕트에 참여한 팀원들이 서로의 task를 이해하기 위해서는 여러 가지 방법이 있을 수 있다. 그중에서 코드 리뷰는 단순히 팀원이 개발하는 도메인에 대한 이해를 넘어서 코드 베이스에서 어떻게 동작하는지 이해할 수 있다. 이것은 팀원이 부재중이더라도 다른 사람이 빠르게 대응할 수 있다는 것을 의미한다.

입사 후에 다른 두 명의 팀원과 함께 백오피스 개발에 참여하게 되었다. 그때는 코드 리뷰를 적극적으로 하지 않고 일정에 쫓겨 각자 개발하기 바빴다. 일부 배포가 된 상태에서 팀원이 휴가를 떠났고 그 팀원이 맡았던 부분에서 오류가 발생했다. 코드 상에서 오류는 쉽게 찾을 수 있었지만 해당 오류를 찾기까지의 코드를 파악하는데 꽤 오랜 시간이 걸렸다. (이때부터 코드 리뷰의 필요성을 서서히 느낄 수 있었다.)

## 2. 코드의 통일성

코드를 작성할 때 개발자들마다 기준이 있고 스타일이 있다. 특히 코드의 규모가 커질수록 여러 명의 개발자들이 규칙을 지키고 일관성 있게 작성하는 것이 추후에 코드를 파악하고 유지 보수하는데 도움이 된다. 하지만 여러 명의 개발자가 하나의 프로덕트를 개발할 때 일관성 있게 코드를 작성하는 것은 어렵다. 파일의 위치, 변수명 등 세세하게 모든 것을 맞추는 것은 어렵다.

이를 사전에 맞추기 위해서 팀원들과 코딩 컨벤션을 정하고 지키도록 노력해야 한다. 그리고 컨벤션을 잘 지키는지 확인하기 위해서는 코드 리뷰가 필요하다. 우리 팀은 한 명이 여러 개의 프로덕트에 참여하여 개발하고 있다. 입사 후 한 달간은 팀에서 진행하고 있는 프로덕트를 파악하는 시간을 가졌다. 그런데 프로덕트마다 코드의 구조가 달랐고 사용하는 라이브러리도 달랐다. A 프로덕트는 상태 관리와 데이터 패칭을 위해 SWR을 사용하고, B 프로덕트는 React-Query를 사용하고 있었다. 그래서 같은 역할을 하는 여러 개의 라이브러리도 공부해야 했기 때문에 파악하는데 어려움이 있었다.

하지만 코드 리뷰를 진행한다면 라이브러리 하나를 사용하려고 할 때도 모든 팀원의 의견을 듣고 사용할지 말지 결정하게 된다. 그렇게 되면 여러 개의 프로덕트에서 동일한 라이브러리를 사용하게 되고 신규 입사자가 프로덕트를 파악하는 것이 조금은 수월해진다. 또한, 하나의 프로덕트 내에서도 마치 한 사람이 코드를 작성한 것처럼 코드 구조와 변수명 등을 통일시켜 전체적인 코드를 파악하는 것이 쉬워진다.

## 3. 개인의 성장

대부분의 직종이 그렇지만 개발자도 성장을 위해 꾸준히 공부할 필요가 있다. 각자 공부를 해야 하는 이유가 다르겠지만 개인적으로는 좋은 서비스를 사람들에게 빠르게 제공하기 위함이다. 하지만 24시간 중 자는 시간, 밥 먹는 시간, 운동하는 시간, 일하는 시간을 제외하면 하루에 공부할 수 있는 시간은 4~5시간 정도이다. 물론 부족하지 않은 시간일 수 있지만 많은 시간을 확보할수록 좋을 것이다.

하루 중 일을 하는데 많은 시간을 할애한다. 그런데 이 시간을 본인이 성장할 수 있는 시간으로 생각한다면 더 많은 시간을 확보할 수 있다. 그중 코드 리뷰 시간이 그렇다. 다른 사람의 코드를 리뷰하다 보면 처음 알게 된 개념을 배울 수 있고, 기존에 잘못 알고 있던 개념을 바로잡을 수 있으며 사고하는 능력이 확장될 수 있다. 그리고 최소한 리뷰어들을 생각하여 가독성 높은 코드를 고민하여 작성하다 보면 본인의 코드 퀄리티가 향상된다.

물론 코드 리뷰를 통해서 스스로 발전하기 위해서는 올바른 코드 리뷰 방법을 가져야 하며 코드 리뷰를 통해서 하나라도 배우려는 본인의 태도가 바탕이 되어야 한다. 그렇기 때문에 성장에 욕심이 있는 사람이라면 코드 리뷰를 하나의 업무라고 생각하지 말고 배울 수 있는 기회라고 생각하면 많은 도움이 될 것이다.

## 마무리

나를 포함한 팀원들 모두가 코드 리뷰의 필요성을 알게 된 이후에는 코드 리뷰를 게을리할 수 없었다. 아무리 촉박한 일정이라도 반드시 코드 리뷰를 거쳤다. 이처럼 꾸준히 코드 리뷰를 한 결과 팀원들이 고민하고 해결하려는 문제의 난이도가 높아졌고, 단순히 컨벤션을 지키지 않았다는 의견을 다는 것을 넘어서 라이브러리를 효과적으로 쓰고 다르게 해결할 수 있는 방법을 제시하기도 한다. 하지만 실질적으로 도움이 되기 위해서는 좋은 방법으로 코드 리뷰를 진행해야 하며 이 내용은 “어떻게 코드 리뷰를 해야 할까?”에서 알아보기로 하자.
